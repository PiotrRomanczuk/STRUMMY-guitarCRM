Recruiter Demo Readiness

2026-07-19 · branch feature/audit-findings-implementation · target strummy.vercel.app → StudentManager (uwh via Cloudflare tunnel)

What was built

Strummy was made presentable as a self-serve demo a recruiter can explore alone: a one-click demo login, a studio full of realistic data, every visible button working (including create/edit flows), and no dead ends or false claims on the public pages.

1. Three real bugs found and fixed

These were breaking the live app for real users, not just the demo:

Bug Symptom Fix
Attaching a song to a lesson failed The song_status_history table was missing, so a trigger on the repertoire cascade aborted the transaction. lesson_songs had 0 rows against 57 lessons — the core loop had never completed on this stack. Migration 20260719190000_restore_song_status_history.sql (DDL + RLS mirrored from the baseline)
Students page was empty for everyone profiles.student_status was missing, so the people-list SELECT errored and the roster rendered “No people match these filters” for teachers and admins alike. Migration 20260719193000_restore_student_status_columns.sql; 61 existing students with lessons marked active
Production build was broken A client component imported helpers from a Server Component, dragging next/headers into the client graph. The branch could not deploy. Helpers extracted to StudentDetailEditorial.shared.tsx

2. Production environment repaired

Six Vercel production variables had a trailing CRLF baked into their values — every cleaned value landed on its exact canonical length (Gmail app password 16 chars, Google client secret 35, PostHog host 24), confirming the corruption. GMAIL_USER, GMAIL_APP_PASSWORD, GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET (SMTP auth and OAuth would reject these) plus the two PostHog vars. NEXT_PUBLIC_APP_URL was also set so email links stop pointing at localhost.

4. Demo accounts can now write

Demo accounts were blocked from mutating at ~90 call sites, mostly failing silently. DEMO_WRITES_ENABLED=true (production only) lifts that block so a recruiter can actually create a lesson or edit a song. Local and CI keep the guard, so the demo-mutation-guard E2E suite still exercises the blocked path. 15 API routes that inlined if (flags.isDevelopment) were rewritten onto the flag-aware helper.

5. Public surface cleaned

  • Landing/auth pages no longer render a second, legacy purple header — which also removes the clickable “Remote DB” debug badge that was firing Supabase probes for anonymous visitors, and a console.log on every navigation.
  • 11 footer/FAQ links that 404'd are gone; social placeholders replaced with the real repo.
  • Both “Watch demo” buttons (which only scrolled to a text section) now open the live demo.
  • FAQ no longer claims students don't need accounts (they do). Pricing is labelled as planned and lists only features that exist.
  • Song detail no longer shows a “Planned” roadmap card on every song.
  • Admin cards no longer contain engineering-log copy (“Real query lands in a follow-up”).
  • PWA icons created — “Install app” now works. Avatar upload hidden in production (this stack has no storage service, so it could only ever fail).
  • CSP now allows the PostHog asset host; it was blocked, erroring on every page load.

6. Navigation opened up

Verified-working features were revealed: Fretboard, AI Assistant and AI Chat for teachers; My Repertoire and a new Practice Log item for students. Stub pages stay hidden, and a new test asserts no nav item points at a “Coming soon” route. The dead DEMO_HIDDEN_ITEMS list (a strict subset of the core-loop list, so it never did anything) was deleted.

How to test it

Fastest path: open https://strummy.vercel.app/sign-in?demo=true — one click lands you in Sarah Mitchell's teacher dashboard, no typing.

Setup

  1. Reseed for a clean, current week: npm run seed:demo — it prints the target host and asks you to type it back.
  2. Local run against the same database: DEMO_WRITES_ENABLED=true npm run build && DEMO_WRITES_ENABLED=true npx next start -p 3100
  3. Credentials — teacher sarah@strummy.app, students emma@ / carlos@ / lily@ / james@strummy.app, all Demo2024!

Teacher walkthrough (expect all to work)

  1. Landing page — exactly one header, no “Remote DB” badge. Click every footer link: none should 404. “Try the live demo” logs you straight in.
  2. Dashboard — today's lessons on the day spine, week-density bars filled, roster and song library cards populated.
  3. Lessons → open a past lesson: teacher notes plus attached songs with status badges. Was broken before — songs could never be attached.
  4. Students → “4 shown” with Emma, Carlos, Lily, James. Open one: stats, preferences and repertoire. Was empty for every role before.
  5. Create a song (Songs → New) — it should save and redirect to the new song, not fail with “not available on test accounts”. Same for a new lesson or assignment.
  6. Songs → open one: chords, key, tempo, lyrics. No “Planned” roadmap card.
  7. Fretboard — pick a key and scale; the diagram updates and the URL carries state.
  8. AI Assistant — send a prompt; it should stream a response (OpenRouter key verified live).
  9. Notifications bell — unread badge; feed shows the daily summary and student activity.
  10. Calendar — renders with a Connect prompt. Do not run the Google connect flow (redirect URI isn't configured for this deployment).

Student walkthrough (sign in as emma@strummy.app)

  1. Sidebar shows My Lessons / My Songs / My Assignments / My Repertoire / Practice Log — and nothing broken.
  2. Repertoire — songs with statuses and her self-ratings; change a rating and it persists.
  3. Practice Log — four weeks of sessions with durations and BPM; log a new one.
  4. Assignments — advance one from not started → in progress; no edit/delete controls (correct for a student).
  5. Notifications — “Lesson tomorrow at 16:00” and the rest of her feed.
  6. Confirm isolation: she must not see other students anywhere.

Edge cases worth a look

  • Mobile at 390px — editorial pages use fixed-width grids; check the dashboard and lists.
  • Five wrong passwords locks an account for 30 minutes. Sign in before a live walkthrough.
  • Re-run npm run seed:demo to reset; it refreshes “this week” relative to today, so run it if the demo data drifts into last week.

Known limitations

  • Google Calendar connect — GOOGLE_REDIRECT_URI is unset in production; the page renders fine but don't run the connect flow.
  • Student emails — STUDENT_EMAILS_ENABLED is off, so nothing is sent to the fake demo addresses. In-app notifications still fire.
  • Avatar upload — hidden in production; the manual URL field still works.
  • Sentry — no DSN configured in production (warns at boot, doesn't fail).
  • Song catalog — 429 songs with 20 duplicated titles; only the 7 demo songs carry full lyrics and chords. Cosmetic, left alone deliberately.
  • Mobile — editorial pages use fixed-width grids with no breakpoints.
