Dead Dashboard Cards Removal

Date: 2026-07-19  ·  Author: Claude  ·  Branch: fix/e2e-desktop-chrome-failures  ·  Follows up: docs/manual-tests/2026-07-18-e2e-stale-test-repair.html

Resolves the one open decision left in the 2026-07-18 stale-test repair report: "Background task spawned: remount or delete TodayLessons / UpcomingLessons (and possibly the dead LegacyShell)." Verdict: delete. Both cards, and the shell that rendered them, were provably unreachable in production.

1. What was actually done

Why delete, not remount

app/dashboard/layout.tsx redirects to /sign-in when there is no authenticated user, and to /onboarding when the user has no role. By the time app/dashboard/page.tsx runs, user is guaranteed non-null and at least one of isAdmin / isTeacher / isStudent is true. resolveActiveView() always resolves to 'admin' | 'teacher' | 'student', and each of those three branches in DashboardPage already returns an editorial view when user is set — so the old LegacyShell fallback (which rendered TodayLessons + UpcomingLessons) could never actually render for a real signed-in user. It was dead code.

Functionally, TeacherDaySpine (inside TeacherDashboardEditorial) already covers what TodayLessons did — a full schedule view of today's lessons, driven by the same getTeacherDayLessons query. There was no editorial equivalent asked for "upcoming this week"; reviving it would have been new feature work on a test-repair branch, not a bug fix.

Change Files
deleted dead components components/dashboard/cards/TodayLessons.tsx, TodayLessons.helpers.ts, UpcomingLessons.tsx, UpcomingLessons.helpers.ts, UpcomingLessons.helpers.test.ts, index.ts (whole components/dashboard/cards/ directory)
deleted orphaned E2E specs tests/e2e/dashboard/today-lessons.spec.ts, tests/e2e/dashboard/upcoming-lessons.spec.ts — both were already test.describe.skip'd pending exactly this decision.
simplified app/dashboard/page.tsx Removed LegacyShell, resolveRoleLabel, and now-unused imports (Card/CardHeader/CardContent/CardTitle, Suspense, LoadingCard, the two deleted card components). The final fallback (reached only if user is unexpectedly null at this point — a defensive edge case, not a real path) now does redirect('/sign-in?redirect=/dashboard'), mirroring the same guard the layout already uses, instead of rendering a "Dashboard rebuild in progress" placeholder that could never be seen.

No product behavior changed for any real user. Every reachable dashboard view (admin / teacher / student editorial) is untouched.
