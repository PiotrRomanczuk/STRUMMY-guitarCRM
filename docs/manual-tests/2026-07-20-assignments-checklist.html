Assignment Checklist & Progress — Manual Test

Date: 2026-07-20 · PR: B of the assignments teacher-POV slice · Branch: feature/assignments-triage

What was built

An assignment can now carry a checklist of sub-tasks, giving a percent-complete progress signal on top of the coarse status:

  • Teacher authors the checklist in the create/edit form (add / edit / reorder / remove, up to 20 items).
  • Student ticks items on the assignment detail page; the Progress card shows a bar and “N / M done · %”.
  • The assignments list shows a compact progress bar per row when a checklist exists.
  • The status state machine is untouched — ticking items never changes status; the student still explicitly marks Start / Complete.
Security (ADR-0001). A student’s only checklist write path is the student_toggle_checklist_item SECURITY DEFINER RPC, which by construction flips exactly one existing item’s done flag — it cannot add, remove, reorder, or edit item text. Teachers/admins author via the existing owned update. This mirrors the ASG-3 status RPC.
DB gate — not yet applied. This PR ships the migration supabase/migrations/20260720000001_assignment_checklist.sql but it has NOT been applied to any live DB in this session (the uwh StudentManager stack is live-prod-conflated per the planner vault, and StrummyProd is prod). Before the UI works end-to-end you must:
  1. Apply the migration to a safe stack (a throwaway/branch DB, or StrummyProd during a zero-user window).
  2. Run the RLS suite there: npm run test:rls (targets lib/testing/rls/__tests__/assignment-checklist.rls.test.ts).
Type generation was intentionally skipped — the server action client is untyped, so no database.types.ts change is required and a regen against a drifted DB was avoided.

How to manually test (after applying the migration)

Dev credentials: teacher p.romanczuk@gmail.com / test123_admin, student emma@strummy.app / Demo2024!.

As a teacher

  1. Create a new assignment. Under Brief, use “+ Add checklist item” to add 3 steps; reorder with ↑/↓; remove one with ×. Save.
  2. On the detail page, confirm the Progress card shows the checklist with a 0% bar.
  3. Open the assignments list — the row shows a compact 0/2-style progress bar.
  4. Edit the assignment; confirm the checklist loads pre-filled and edits persist.
  5. Optional: as the teacher, tick an item on the detail page — it should update (owner path).

As the student

  1. Open the assignment. Tick a checklist item — the bar advances and “N / M done” updates.
  2. Reload — the tick persisted.
  3. Confirm ticking every item does NOT auto-complete the assignment — you still press the explicit “Mark complete” button.

Security spot-check (optional, requires a session token)

  • A direct PostgREST PATCH to /rest/v1/assignments from a student session that tries to rewrite checklist text must leave the row unchanged (covered by the RLS suite).
