Strummy — Minimal Working Core Loop

Date: 2026-07-17 · Branch: feature/minimal-core-loop · Author: Claude

A trust pass, not a feature pass: prove the basic core loop works reliably on the local database for two real actors (Piotr as admin/teacher + one student), shrink the visible surface to just that loop, and back it with green end-to-end tests. Everything else is hidden from the sidebar (not deleted) until proven, step by step.

What was actually done

Area Change & resulting behavior
Sidebar trimmed components/navigation/menuConfig.ts — an always-on CORE_LOOP_HIDDEN_ITEMS filter reduces the nav to Dashboard · Lessons · Songs · Assignments · Students · Settings for teacher/admin, and Dashboard · My Lessons · My Songs · My Assignments · Settings for students. Theory, Skills, Health, Analytics, Calendar, Fretboard, AI, Stats, Repertoire are hidden from nav (routes still reachable by URL — nav-only, reversible).
Sign-in strand fix app/(auth)/sign-in/page.tsx — the initial auth check awaited supabase.auth.getUser() with no timeout/catch, so a hung call (supabase-js navigator.locks, common under dev HMR) stranded the page on "Loading…" forever. Now races a 3s timeout and falls through to the form on any hang/error.
Shadow lifecycle proven Admin creates a student as an unclaimed shadow at /dashboard/users/new → "Invite to claim" → the handle_new_user trigger migrates the student's lessons + assignments onto the real account, deletes the shadow row, keeps one profile. Verified end-to-end (UI + DB). Real invite email delivered to Gmail with a redirect_to=localhost:3000/accept-invitation link.
Realistic seed data Parsed a WhatsApp lesson export into 10 dated lessons + 20 songs and seeded them onto the test student (romanczukpiotr95), deduping songs against the existing library. The student now shows a real Dec 2025 → Mar 2026 history.
E2E hardening New tests/helpers/seed-ids.ts resolves student/teacher/admin ids from the configured emails at runtime (replacing stale hard-coded UUIDs); student specs target the seeded item deterministically instead of .first(). New __tests__/components/navigation/menuConfig.test.ts locks the trimmed nav.

The core-loop journeys (all verified)

  1. Auth — admin/teacher sign-in, student sign-in, sign-out, page refresh (no loop).
  2. Shadow → invite → claim — admin creates shadow; attaches lesson + assignment; invites; student sets password + logs in; references migrate onto the real account; shadow row gone; one profile per email.
  3. Songs CRUD (teacher) — create → view → edit → soft-delete (repertoire cascade).
  4. Lessons CRUD (teacher) — create (attach a song) → view → edit → soft-delete.
  5. Assignments CRUD (teacher) — create → view → edit → soft-delete.
  6. Student surface — sees only their own lessons/songs/assignments; updates assignment status; cannot read another student's data (RLS).
  7. Scope — sidebar shows only the core items per role.

How to run & manually test

Prerequisites (Piotr's setup, on the Orange LAN):
  1. Local Supabase (StudentManager stack) is up on uwh. It's reachable LAN-direct — no Tailscale, no SSH tunnel needed. Node fetch to http://192.168.1.75:54321 works.
  2. .env.local has NEXT_PUBLIC_SUPABASE_LOCAL_URL=http://192.168.1.75:54321 + the local anon & service-role keys.
  3. Build once and run the production server (dev/Turbopack HMR is flaky on this repo's slow filesystem): 
    npm run build && PORT=3000 npm run start
    Confirm db: "local" in the logs.
Manual walk-through
  1. Open http://localhost:3000/sign-in. Sign in as p.romanczuk@gmail.com / test123_admin. Confirm the "Local DB" badge and the trimmed sidebar (Dashboard, Lessons, Songs, Assignments, Students, Settings only).
  2. Create a student: Students → + New student, enter name + invite email → the student appears as Unclaimed.
  3. Create a song (Songs → New), a lesson (Lessons → New, attach the song), and an assignment (Assignments → New) for that student. Edit and delete each to confirm the full CRUD cycle.
  4. On the student detail page, click Invite to claim → Send. The shadow's data migrates to the real account.
  5. Sign out, sign in as the student, and confirm they see their lessons/assignments and the minimal student sidebar.
Automated (E2E) — run serially 
npx playwright test tests/e2e/dashboard/sidebar.spec.ts \
  tests/e2e/auth/role-login.spec.ts tests/e2e/auth/sign-out.spec.ts \
  tests/e2e/teacher/{songs,lessons,assignments}-crud.spec.ts \
  tests/e2e/student/{songs-read,lessons-read,assignments-interact}.spec.ts \
  --project="Desktop Chrome" --workers=1
Expected: 25 passed / 0 failed / 5 skipped. Plus the simplest sanity check: 
curl -s "$BASE/rest/v1/lessons?student_id=eq.<id>&select=title,status,scheduled_at" -H "apikey: $SRV" -H "Authorization: Bearer $SRV"

Known limitations (out of scope for this pass)

  • E2E suite needs --workers=1 / session reuse: it shares a single test student, so parallel runs collide on the per-(teacher,student) lesson-number trigger, and rapid re-logins trip the auth rate limiter. Making it parallel-safe (per-worker students) is a follow-up.
  • tests/e2e/auth/sign-up-complete.spec.ts has 10 stale-selector failures (self-signup form) — not a core journey (students arrive by invite); left for a separate pass.
  • Hidden features are nav-only; their routes are still reachable by direct URL (no route-guarding, by design).
  • Local GoTrue (StudentManager) uses real Gmail SMTP; the emailed claim link verifies via the Cloudflare tunnel to the same stack then redirects to localhost:3000. Local claim via the emailed link works; automated tests set the password via the admin API.
